Quickstart

          

            From sealed-box Acuity to robot code reading AprilTag
            poses — or running custom compute the RoboRIO doesn't
            have headroom for — in about ten minutes.
          

          
    
            
  1. Power Acuity (USB-A from the RoboRIO is fine, or any
                5 V / 1 A source).
    2. 
            
  2. Wired to the radio (competition):
                plug eth0 into the radio, power on. Acuity reads its
                own team number off the DHCP lease and shows up in
                Manager automatically — no captive portal needed.
                
    
                Wireless setup: connect a phone to
                the open WiFi Acuity-Setup-XXXX, then
                hit the captive portal (or browse to
                http://192.168.50.1/). Fill in team
                number + robot WiFi SSID + password.
    3. 
            
  3. Open this Manager — your device shows up in the
                Devices tab automatically (mDNS
                discovery, IP-agnostic).
    4. 
            
  4. Open the Libraries tab, pick your
                team's language, and point the picker at your robot
                project folder. Manager drops a single
                AcuityClient helper file into your
                project — no vendordep, no pip dep.
    5. 
            
  5. Add 3 lines to your robot code (see your language's
                section below) and you're reading AprilTag data.
    6. 
          

        


        
        

          
How it fits together

          

            Acuity is two things in one box. The default pipeline
            does vision; the rest of the device is a general-purpose
            Linux coprocessor on the robot's network that your team
            can put real compute on — high-rate pose estimation,
            trajectory pre-computation, interception math, scouting
            lookups, anything CPU-bound the RoboRIO doesn't have
            cycles for.
          

          

            
The device — vision

            

              Camera → OpenCV capture → pyapriltags →
              /acuity/* NT4 topics. Serves a web
              dashboard at :8080 (live feed, settings,
              network controls).
            

            
The device — coprocessor

            

              Drop any Python script into
              /var/lib/acuity/scripts/; run it on
              demand via /api/scripts/<name>/run
              or at boot via a systemd unit. Publish results to
              /acuity/user/* NT topics. See
              Coprocessor recipes for
              concrete worked examples (high-rate pose fusion,
              trajectory pre-compute, intercept prediction,
              scouting lookup).
            

            
The AcuityClient helper file

            

              Single self-contained file Manager drops into your
              robot project. Wraps NT4 with a typed
              getBestTag() →
              Optional<TagDetection> API and
              tracks heartbeat staleness so you can tell "no tag in
              view" from "device offline". Zero external
              dependencies — no vendordep, no Maven artifact, no
              PyPI package. Just a file in your source tree.
            

            
This Manager

            

              Driver-station companion. Discovers devices,
              one-clicks firmware updates, opens an in-app SSH
              terminal, drops the helper file into your project,
              and embeds the camera feed (Camera tab).
            

          

          

            Robot code talks to Acuity via NetworkTables 4. It
            never opens an HTTP socket to the device — it reads
            NT4 topics like every other component on the bus. The
            REST endpoints below are for tools (Manager, dashboard,
            scouting scripts, your own laptop), not robot code.
          

        


        
        

          
Hardware

          

            Acuity is sold as a unit, not a board. Each device is
            built for speed and configured for the camera count and
            workload it's going to run. From robot code's
            perspective the device looks the same regardless of
            what's inside — same hostname, same NT4 topics, same
            AcuityClient helper — so you write against
            the contract, not the silicon underneath.
          


          
Architecture

          

            Acuity normally runs on the Raspberry Pi or Orange Pi
            architecture family, picked per device to fit the
            workload. The firmware build, the API surface, and the
            user experience are identical across the tiers — only
            the silicon underneath varies.
          


          
Specs are not set in stone

          

            We don't publish a single fixed board spec. The compute
            tier, memory, and peripheral mix we ship may change
            between batches as parts availability moves and as we
            tune for what teams are actually running.
            These changes don't impact price.
          

          

            Regardless of which board is inside, every Acuity
            performs at extremely similar speeds in actual use.
            That parity isn't an accident — it's because every
            device runs the same heavily-optimized Linux
            build, a lightweight distribution we maintain
            based on Raspberry Pi OS / Debian. Vision pipeline,
            kernel tunings, service set, network stack, and storage
            layout are identical across tiers; only the silicon
            underneath varies.
          

          

            Need exact specs for a specific use case (e.g. you're
            sizing a heavier custom workload, or your inspector
            wants a part-number sheet)? Contact us
            and we'll send the bill of materials for the unit you
            received.
          


          
Multi-camera, tuned at the hardware level

          

            Multiple-camera setups are where most off-the-shelf FRC
            vision boards fall over — they advertise the camera
            count and quietly degrade once you actually run all of
            them. Acuity handles this differently:
          

          
    
            
  • We pick a compute module with enough I/O bandwidth
                for the camera plan (CSI lanes, dedicated USB
                controllers, separate buses for each sensor).
    • 
            
  • The firmware image is tuned per device — kernel
                buffer sizes, CPU governors, USB and CSI driver
                parameters — to match the cameras we built it
                with.
    • 
            
  • We do hardware-level modifications where it matters:
                power rail capacity, thermal pads, antenna placement,
                additional storage, swap configuration. The board
                you receive isn't a stock module with a sticker;
                it's a sub-assembly we stand behind for the workload
                we sold you.
    • 
          

          

            Net effect for the robot programmer: you can ask Acuity
            to track AprilTags, detect game pieces, fuse a swerve
            pose at 200 Hz, and log telemetry concurrently — and
            the device that arrives is provisioned to do all of it
            without you tuning anything.
          


          
Adding cameras at runtime

          

            The primary camera always exists and feeds the AprilTag
            pipeline. Additional cameras are configured via the
            extra_cameras field on the device's
            /api/settings — one entry per camera, each
            with its own name, index,
            capture + stream knobs:
          

          

            
curl -X POST http://acuity-1279.local:8080/api/settings \
  -H 'Content-Type: application/json' \
  -d '{
    "extra_cameras": [
      { "name": "intake", "index": 1,
        "resolution":        [640, 480],
        "target_fps":        15,
        "stream_resolution": [320, 240],
        "stream_quality":    50,
        "stream_max_fps":    15 }
    ]
  }'

            
          

          

            GET /api/cameras lists every active camera
            (primary + extras) with live FPS, resolution, and
            connection state. The MJPEG stream picks a camera via
            ?cam=<name>:
            http://acuity-1279.local:8080/stream.mjpg?cam=intake.
            Same query works on /api/snapshot.
          

          

            Manager's Camera tab auto-shows a
            camera picker the moment a device reports more than one
            camera from /api/cameras — there's nothing
            to configure on the laptop side.
          

          

            The defaults applied to extras (lower resolution, lower
            JPEG quality, capped FPS) are the auto-quality safety
            net: adding a second camera at the click of a button
            shouldn't accidentally double the capture CPU. Override
            them per-camera once you know what your specific setup
            can sustain.
          


          
Storage — at least 64 GB on every device

          

            Every Acuity ships with at least 64 GB of
            storage, regardless of compute tier. That's
            roughly 128× the available storage on a
            RoboRIO, which is the practical reason teams
            can:
          

          
    
            
  • Keep a full season's worth of
                telemetry logs on the
                device, ready to scp off post-event.
    • 
            
  • Park scouting CSVs, alliance tendency tables, and
                playbook configs in /var/lib/acuity/
                without worrying about size.
    • 
            
  • Cache trajectory libraries, pre-computed lookup
                tables, and recorded match video for review.
    • 
            
  • Snapshot the entire system before a competition and
                roll back from the same SD card if a mid-event
                change misbehaves.
    • 
          

          

            Storage is the resource you'll feel first on the RoboRIO
            when you try to put real logging or pre-computed assets
            on the robot. Acuity exists in part so you don't have to.
          


          
Status light

          

            The onboard activity LED is taken over by Acuity at boot
            so a glance at the device tells you which state it's in.
            No laptop required. Patterns from "I'm waiting for setup"
            to "I crashed, please help" are visible across a pit.
          

          

            
PatternStateMeaning

            
              
Slow double-pulsebootSystem booting; first-boot script still running.

              
Rapid blink (5 Hz)apAP-mode setup wizard up. Connect to Acuity-Setup-XXXX and configure.

              
Medium blink (~1 Hz)sta-connectingJoining the configured team WiFi.

              
Solid onconnectedLive on team WiFi. Dashboard reachable.

              
Short-short-long (every ~3 s)wiredLive on ethernet. Visually distinct from connected so an inspector can tell at a glance which link is in use.

              
Triple-blip + pauseupdateFirmware update in progress. Don't power-cycle.

              
Morse SOS  · · · — — — · · ·errorA critical service hit systemd's failure limit (acuity-firstboot crashed, acuity-dashboard refused to come up, etc.). SSH in or pop the SD card.

              
Solid offoffForced dark — set manually for night-match stealth.

            
          

          

            The state is published to /run/acuity/status
            (a single word on one line). Custom scripts can drive it:
          

          

            
acuity-status update   # before a long-running operation
acuity-status connected # back to normal
acuity-status off       # silence the LED entirely

            
          

          

            Pre-userspace boot failures (kernel or start.elf not
            found, SD-card unreadable, …) are signalled by the GPU
            firmware's own blink codes — those run before any of our
            services can take over the LED. The error
            state above covers everything that breaks after
            Acuity has taken control.
          


          
Tested where it matters

          

            Every Acuity tier is tested on our own team's robots
            before it ships. That's not a focus group — it's the
            same hardware running the same season, with the same
            wear, the same radio environment, and the same "the
            match starts in ten minutes" pressure your team is going
            to put on it.
          

          

            On top of that we bench-test each tier against the
            options teams realistically consider:
          

          
    
            
  • Limelight on the vision-only path:
                AprilTag detection range, latency, and stability under
                radio contention.
    • 
            
  • A base Raspberry Pi (or comparable
                stock board) running off-the-shelf FRC firmware: how
                much you have to fight the OS to get the same
                workload running, and how it holds up under
                multi-camera + non-vision loads.
    • 
          

          

            If a tier doesn't beat both on the workloads we ship it
            for, we don't ship it.

Minimal usage

          

            
import frc.robot.acuity.AcuityClient;
import edu.wpi.first.wpilibj.smartdashboard.SmartDashboard;

public class Robot extends TimedRobot {
  private final AcuityClient acuity = AcuityClient.getInstance();

  @Override
  public void teleopPeriodic() {
    acuity.getBestTag().ifPresent(tag -> {
      SmartDashboard.putNumber("acuity/id",         tag.id());
      SmartDashboard.putNumber("acuity/distance_m", tag.distanceMeters());
      SmartDashboard.putNumber("acuity/yaw_deg",    tag.yawDeg());
    });
  }
}

TagDetection fields

          

            TagDetection is a record, so
            field access is parens-style (tag.id(),
            tag.distanceMeters()).
          

          

            
AccessorTypeMeaning

            
              
id()intAprilTag ID. -1 means "no tag" (you'll never see this through Optional; raw NT4 readers will).

              
distanceMeters()doubleDistance from camera to tag center.

              
yawDeg()doubleCamera-to-tag yaw, signed. Negative = left of center.

              
pitchDeg()doubleCamera-to-tag pitch, signed.

              
tx() / ty()doublePixel offset from frame center, normalized to [-1, 1] (resolution-independent).

              
area()doubleFraction of frame the tag occupies, 0..1.

              
timestamp()doubleFPGA seconds when the source frame was captured.

              
decisionMargin()doublepyapriltags confidence — bigger = stronger detection.

            
          


          
Common patterns

          
Auto-aim with the best tag

          

            
acuity.getBestTag().ifPresent(tag -> {
  double headingErrorDeg = tag.yawDeg();
  double turnPower = -kP_HEADING * headingErrorDeg;
  drivetrain.arcadeDrive(0, turnPower);
});

            
          


          
Filter to specific tag IDs

          

            Easier path: set the preferred-tag-ID list in the
            dashboard's Settings tab — Acuity then picks "best" only
            from those IDs and you read
            getBestTag() as usual. Or read all visible
            tags directly off NT4 if you want client-side filtering;
            see the NT schema's
            tags/all string topic.
          


          
Handle the device dropping mid-match

          

            getBestTag() returns empty whenever the
            heartbeat is stale (>250 ms). Use
            isConnected() to distinguish "no tag in
            view" from "device unreachable":
          

          

            
if (!acuity.isConnected()) {
  // Device is gone — fall back to encoders + IMU only.
  setVisionTrustWeight(0.0);
} else {
  setVisionTrustWeight(1.0);
}

Common patterns

          

            
# Drop in commands/auto_aim.py
import commands2
from acuity_client import AcuityClient

class AutoAim(commands2.Command):
    def __init__(self, drivetrain):
        super().__init__()
        self.drive = drivetrain
        self.acuity = AcuityClient()
        self.addRequirements(drivetrain)

    def execute(self):
        tag = self.acuity.get_best_tag()
        if tag is None:
            self.drive.arcadeDrive(0, 0)
            return
        turn = -0.04 * tag.yaw_deg
        self.drive.arcadeDrive(0, turn)

    def isFinished(self):
        tag = self.acuity.get_best_tag()
        return tag is not None and abs(tag.yaw_deg) < 1.5

Coprocessor endpoints

          

            Acuity is sold as a vision device, but it's a small
            Linux board on the robot's network — i.e. the only
            CPU + Python runtime no other component on the bus has.
            The endpoints below let you lean on it as a general
            coprocessor: get system stats, restart services, run
            user scripts, publish arbitrary values to NetworkTables.
          


          
System info

          

            
EndpointReturns

            
              
GET /api/systemFull system snapshot — model, kernel, CPU%, RAM%, disk%, SoC °C, uptime, eth0/wlan0 IPs, camera status, scripts dir.

            
          

          

            Cheap (<10 ms — built from /proc +
            /sys reads). Anything that needs sustained
            polling should subscribe to NT4's
            /acuity/health/* topics instead.
          


          
Service control

          

            
EndpointReturns

            
              
GET /api/servicesList of controllable units + their current systemd state.

              
POST /api/services/<name>{ "action": "restart" } (also start / stop / status). Whitelisted units only.

            
          

          

            Restarting acuity-dashboard.service kills
            the process answering your request — the response will
            not reach the client. Manager handles the dropped
            connection by treating it as success.
          


          
Script runner

          

            Drop any executable file (or .py) into
            /var/lib/acuity/scripts/ on the device and
            run it from any HTTP client. Each run gets a buffered
            stdout/stderr you can poll until exit. The easiest way
            to manage these is Manager's Scripts
            tab — pick your device, edit in-app, hit Run, watch
            output. The REST endpoints below are what the tab
            drives:
          

          

            
EndpointReturns

            
              
GET /api/scriptsList scripts in the scripts dir (name, size, mtime, executable bit).

              
GET /api/scripts/<name>Plain-text source — Manager's editor opens with this.

              
PUT /api/scripts/<name>Body is the raw source. Creates or overwrites. Non-.py files get +x on write.

              
DELETE /api/scripts/<name>Remove the script. Any schedule pointing at it is purged too.

              
POST /api/scripts/<name>/runOptional body { "args": ["..."] }. Returns a run_id.

              
GET /api/scripts/runsList every run from this dashboard process.

              
GET /api/scripts/runs/<run_id>Run state, exit code, full captured output (capped at 256 KB).

            .py files are launched with

Scheduler

          

            Fire a script automatically on an interval — heartbeats,
            log rotation, periodic snapshots. The dashboard runs a
            built-in scheduler thread that wakes every second; the
            minimum schedulable interval is 5 seconds. Manager's
            Scripts tab has a UI; the REST surface is:

package frc.robot.acuity;

import java.net.URI;
import java.net.http.HttpClient;
import java.net.http.HttpRequest;
import java.net.http.HttpResponse;
import java.time.Duration;

public final class AcuityCommand {
  private static final HttpClient client =
      HttpClient.newBuilder()
        .connectTimeout(Duration.ofMillis(500))
        .build();

  // Best to call from a non-periodic context (button binding,
  // command initialize). Don't put this in a 50 Hz loop.
  public static void applyPipeline(String host, String name) {
    fireAndForget(host, "/api/pipelines/" + name + "/apply", "");
  }

  public static void setCrosshair(String host, double x, double y) {
    fireAndForget(host, "/api/settings",
        "{\"crosshair_x\":" + x + ",\"crosshair_y\":" + y + "}");
  }

  public static void setSelectedTag(String host, int id) {
    fireAndForget(host, "/api/target", "{\"id\":" + id + "}");
  }

  public static void log(String host, String key, double value) {
    fireAndForget(host, "/api/log",
        "{\"key\":\"" + key + "\",\"value\":" + value + "}");
  }

  private static void fireAndForget(String host, String path, String body) {
    HttpRequest req = HttpRequest.newBuilder()
        .uri(URI.create("http://" + host + ":8080" + path))
        .timeout(Duration.ofMillis(800))
        .header("Content-Type", "application/json")
        .POST(HttpRequest.BodyPublishers.ofString(body))
        .build();
    client.sendAsync(req, HttpResponse.BodyHandlers.discarding());
  }
}

from __future__ import annotations
import json, urllib.request

class AcuityCommand:
    def __init__(self, host: str = "acuity.local"):
        self.base = f"http://{host}:8080"

    def _post(self, path: str, body: dict | None = None) -> dict:
        data = json.dumps(body or {}).encode()
        req = urllib.request.Request(
            self.base + path, data=data,
            headers={"Content-Type": "application/json"},
        )
        with urllib.request.urlopen(req, timeout=0.8) as r:
            return json.loads(r.read())

Crosshair calibration

          

            The launcher / arm / shooter on a robot is rarely
            mechanically aligned with the camera's optical axis.
            Acuity's crosshair lets you teach it where "aim point" is
            in normalized image coordinates; from then on,
            yaw_deg and pitch_deg are
            offsets from that point, not from the optical
            center. Robot code drives a heading PID straight off the
            topic and hits where you calibrated.
          


          
Calibrate

          
    
            
  1. Sit the robot at a known distance from a tag and
                fire one shot.
    2. 
            
  2. Note where the projectile actually landed relative
                to the image center, in normalized
                [-1, 1] coords (right is positive
                x, down is positive y).
    3. 
            
  3. Save those as the crosshair:
    4. 
          

          

            
curl -X POST http://acuity-1279.local:8080/api/settings \
  -H 'Content-Type: application/json' \
  -d '{ "crosshair_x": 0.05, "crosshair_y": -0.08 }'

            
          

          

            Affects both the AprilTag and the color-blob outputs.
            Bake different crosshairs into different pipelines if
            your near-shot vs. long-shot calibration differs.
          

          

            Crosshair defaults to (0, 0) — the optical
            center. If you don't calibrate, you get the same
            behavior as before this feature existed.

Sensor fusion patterns

          

            Acuity hands you a pose estimate from AprilTags. Your
            robot has wheel odometry and an IMU. Fusing them gives
            you a pose that's better than either alone. You do the
            fusing on the RoboRIO (where the swerve / pose estimator
            already lives) — Acuity's job is to publish a clean
            measurement and a covariance hint.
          


          
WPILib pose estimator (drop-in)

          

            SwerveDrivePoseEstimator already takes
            vision measurements with a covariance. Wire Acuity into
            it directly:
          

          

            
import edu.wpi.first.math.estimator.SwerveDrivePoseEstimator;
import edu.wpi.first.math.geometry.*;
import edu.wpi.first.wpilibj.Timer;

private final SwerveDrivePoseEstimator estimator = /* ... */;

@Override
public void robotPeriodic() {
  // 1. Wheel + gyro update — every loop.
  estimator.update(gyro.getRotation2d(), modulePositions());

  // 2. Vision update — only when Acuity has a fresh tag.
  acuity.getBestTag().ifPresent(tag -> {
    // Convert tag-relative pose to field-relative using the
    // 2025 AprilTagFieldLayout's known tag pose.
    Pose2d fieldPose = tagToFieldPose(tag);
    // Distance-aware covariance: trust close-range tags more.
    double xyStd  = 0.1 + 0.5 * tag.distanceMeters();
    double yawStd = Math.toRadians(2 + 5 * tag.distanceMeters());
    estimator.addVisionMeasurement(
        fieldPose,
        Timer.getFPGATimestamp() - 0.020,    // ~20 ms latency
        VecBuilder.fill(xyStd, xyStd, yawStd));
  });
}

            
          

          

            tag.timestamp() from the helper is the
            FPGA seconds at the source frame, so you can refine the
            0.020 latency hint if you want lower
            measurement noise.

Lead / intercept calculation

          

            Auto-aim while moving. Pure robot-side math; Acuity just
            reports where the target is now.
          

          

            
// Lead a moving target with a finite-speed projectile.
acuity.getBestTag().ifPresent(tag -> {
  double dist = tag.distanceMeters();
  double tof = dist / SHOOTER_EXIT_MPS;        // time-of-flight
  // Robot velocity in tag frame:
  Translation2d v = drivetrain.getChassisSpeeds()
      .toRobotRelative()
      .toTranslation()
      .rotateBy(Rotation2d.fromDegrees(-tag.yawDeg()));
  double leadDeg = Math.toDegrees(Math.atan2(v.getY() * tof, dist));
  double aimYaw  = tag.yawDeg() + leadDeg;
  drivetrain.setHeadingSetpoint(aimYaw);
});

Recipe 1: High-rate pose fusion

          

            The RoboRIO can run SwerveDrivePoseEstimator
            but it's sharing CPU with everything else on the bus. If
            you want a 200 Hz fused pose (vs. the 50 Hz the periodic
            loop gives you), offload it. Acuity reads odometry +
            heading off NT, fuses with its own tag observations,
            publishes back at 200 Hz.
          

          
On the device: scripts/pose_fuser.py

          

            
"""Complementary-filter pose fuser at 200 Hz.

Reads:  /robot/odom/{x,y,heading_deg}   robot publishes at 50 Hz
        /acuity/tags/best/{distance_m, yaw_deg, id}  (vision)
Writes: /acuity/user/pose/{x,y,heading_deg,fresh}
"""
import math, time, ntcore

inst = ntcore.NetworkTableInstance.getDefault()
inst.startClient4("acuity-pose-fuser")
inst.setServerTeam(0)  # set your team number here

robot   = inst.getTable("robot/odom")
vision  = inst.getTable("acuity/tags/best")
out     = inst.getTable("acuity/user/pose")

# Subscribers
sub_x   = robot.getDoubleTopic("x").subscribe(0.0)
sub_y   = robot.getDoubleTopic("y").subscribe(0.0)
sub_h   = robot.getDoubleTopic("heading_deg").subscribe(0.0)
sub_tid = vision.getIntegerTopic("id").subscribe(-1)
sub_td  = vision.getDoubleTopic("distance_m").subscribe(0.0)
sub_ty  = vision.getDoubleTopic("yaw_deg").subscribe(0.0)

pub_x   = out.getDoubleTopic("x").publish()
pub_y   = out.getDoubleTopic("y").publish()
pub_h   = out.getDoubleTopic("heading_deg").publish()
pub_f   = out.getBooleanTopic("fresh").publish()

# State
fx, fy, fh = 0.0, 0.0, 0.0
ALPHA = 0.92        # odometry trust; (1-alpha) → vision pull
LOOP_HZ = 200
dt = 1.0 / LOOP_HZ

# Known tag field positions (fill in for your field season).
TAG_FIELD = {7: (8.31, 1.45, 0.0)}

next_t = time.monotonic()
while True:
    next_t += dt

    ox, oy, oh = sub_x.get(), sub_y.get(), sub_h.get()
    tid = sub_tid.get()

    # Dead-reckon: trust odometry (cheap, smooth).
    fx, fy, fh = ox, oy, oh

    # When a known tag is in view, pull toward where it implies we are.
    if tid in TAG_FIELD:
        tx, ty, _ = TAG_FIELD[tid]
        # Reduce tag bearing to field-frame XY using current heading guess.
        d  = sub_td.get()
        ya = math.radians(sub_ty.get() + fh)
        tag_implied_x = tx - d * math.cos(ya)
        tag_implied_y = ty - d * math.sin(ya)
        fx = ALPHA * fx + (1 - ALPHA) * tag_implied_x
        fy = ALPHA * fy + (1 - ALPHA) * tag_implied_y

    pub_x.set(fx); pub_y.set(fy); pub_h.set(fh)
    pub_f.set(tid >= 0)

    # Stable 200 Hz loop. time.sleep until next tick.
    slack = next_t - time.monotonic()
    if slack > 0:
        time.sleep(slack)

Recipe 2: Trajectory pre-compute

          

            PathPlanner / Choreo trajectories are fast to generate
            but the RoboRIO would rather not do it mid-auto. Push
            the planning to Acuity over HTTP — robot POSTs a goal,
            Acuity returns waypoints, robot drives them.
          

          
On the device: scripts/plan_path.py

          

            
"""Synthesizes a cubic-Hermite spline between current pose and
goal pose, returns waypoints. Called from robot code via
/api/scripts/plan_path/run with body {"args": ["x0","y0","h0","x1","y1","h1"]}.
Output appears on the run's stdout — robot polls
/api/scripts/runs/<id> until exit_code is non-null."""
import sys, json, math

x0,y0,h0,x1,y1,h1 = (float(a) for a in sys.argv[1:7])

# Tangent magnitude scaled with distance — keeps the spline
# from looping back on short paths.
dist = math.hypot(x1-x0, y1-y0)
m = max(0.5, dist * 1.2)
tx0, ty0 = m*math.cos(math.radians(h0)), m*math.sin(math.radians(h0))
tx1, ty1 = m*math.cos(math.radians(h1)), m*math.sin(math.radians(h1))

N = 40
waypoints = []
for i in range(N + 1):
    t = i / N
    h00 = 2*t**3 - 3*t**2 + 1
    h10 = t**3 - 2*t**2 + t
    h01 = -2*t**3 + 3*t**2
    h11 = t**3 - t**2
    x = h00*x0 + h10*tx0 + h01*x1 + h11*tx1
    y = h00*y0 + h10*ty0 + h01*y1 + h11*ty1
    waypoints.append([round(x, 4), round(y, 4)])

print(json.dumps({"waypoints": waypoints, "count": len(waypoints)}))

Recipe 3: Game-piece intercept prediction

          

            The color-blob detector publishes object centroids at
            ~30 Hz. To intercept a moving piece, you need its
            velocity, which means tracking across frames.
            That's Kalman-filter territory — fits perfectly on Acuity.
          

          
On the device: scripts/intercept.py

          

            
"""Tracks the best color-blob across frames, estimates velocity
via exponential smoothing, projects intercept N seconds ahead.

Reads:  /acuity/objects/best/{tx, ty, area}
Writes: /acuity/user/intercept/{tx_pred, ty_pred, vx, vy, valid}
"""
import time, ntcore

LOOK_AHEAD_S = 0.30   # how far in the future to project
ALPHA = 0.6           # velocity smoothing
LOOP_HZ = 60

inst = ntcore.NetworkTableInstance.getDefault()
inst.startClient4("acuity-intercept")
inst.setServerTeam(0)

obj = inst.getTable("acuity/objects/best")
sub_tx   = obj.getDoubleTopic("tx").subscribe(0.0)
sub_ty   = obj.getDoubleTopic("ty").subscribe(0.0)
sub_area = obj.getDoubleTopic("area").subscribe(0.0)

out = inst.getTable("acuity/user/intercept")
pub_tx = out.getDoubleTopic("tx_pred").publish()
pub_ty = out.getDoubleTopic("ty_pred").publish()
pub_vx = out.getDoubleTopic("vx").publish()
pub_vy = out.getDoubleTopic("vy").publish()
pub_ok = out.getBooleanTopic("valid").publish()

last_tx = last_ty = None
last_t = time.monotonic()
vx = vy = 0.0
dt = 1.0 / LOOP_HZ

while True:
    t = time.monotonic()
    area = sub_area.get()
    if area < 0.002:                   # nothing in view → invalidate
        pub_ok.set(False)
        last_tx = last_ty = None
        time.sleep(dt); continue

    tx, ty = sub_tx.get(), sub_ty.get()
    if last_tx is not None:
        elapsed = max(1e-3, t - last_t)
        nvx = (tx - last_tx) / elapsed
        nvy = (ty - last_ty) / elapsed
        vx = ALPHA * vx + (1 - ALPHA) * nvx
        vy = ALPHA * vy + (1 - ALPHA) * nvy

    pub_tx.set(tx + vx * LOOK_AHEAD_S)
    pub_ty.set(ty + vy * LOOK_AHEAD_S)
    pub_vx.set(vx); pub_vy.set(vy)
    pub_ok.set(True)
    last_tx, last_ty, last_t = tx, ty, t
    time.sleep(dt)

Recipe 4: Scouting / strategy helper

          

            Scouting data lives in a CSV that's too big to bundle in
            robot code (and too easy to forget to deploy). Park it on
            Acuity: scp the CSV up, drop a small lookup script in,
            robot publishes "current opponents" and reads back
            "recommended priority targets."
          

          
On the device: scripts/scout_lookup.py

          

            
"""Reads opponent alliance from robot, reads scouting CSV, writes
back a comma-joined list of recommended priority target IDs.

CSV format: team,priority_tags
            254,7,4
            148,4,7,8
"""
import csv, time, ntcore
from pathlib import Path

CSV_PATH = Path("/var/lib/acuity/scouting.csv")
inst = ntcore.NetworkTableInstance.getDefault()
inst.startClient4("acuity-scout")
inst.setServerTeam(0)

robot = inst.getTable("robot/match")
out   = inst.getTable("acuity/user/scout")
sub_opp = robot.getStringTopic("opponents").subscribe("")
pub_tgt = out.getStringTopic("priority_tags").publish()

def load_tendencies() -> dict[int, list[int]]:
    if not CSV_PATH.exists(): return {}
    d = {}
    with CSV_PATH.open() as f:
        for row in csv.reader(f):
            if not row: continue
            d[int(row[0])] = [int(x) for x in row[1:]]
    return d

tend = load_tendencies()
last_mtime = CSV_PATH.stat().st_mtime if CSV_PATH.exists() else 0
last_opp = ""

while True:
    # Hot-reload on CSV change.
    if CSV_PATH.exists():
        mt = CSV_PATH.stat().st_mtime
        if mt != last_mtime:
            tend = load_tendencies(); last_mtime = mt

    opp = sub_opp.get()
    if opp != last_opp:
        targets: list[int] = []
        for team_s in opp.split(","):
            team_s = team_s.strip()
            if team_s.isdigit():
                targets += tend.get(int(team_s), [])
        # Deduplicate while preserving order.
        seen, ordered = set(), []
        for t in targets:
            if t not in seen: seen.add(t); ordered.append(t)
        pub_tgt.set(",".join(str(x) for x in ordered))
        last_opp = opp

    time.sleep(0.25)   # this isn't latency-critical

            (or filters tags client-side). When the scouting team
            updates the CSV mid-event, just scp the new file —
            the script hot-reloads.

[Unit]

After=acuity-dashboard.service

WantedBy=multi-user.target

acuity.conf options

          

            Per-device config on the FAT32 boot partition at

            any laptop without booting the Acuity device.
          

          

            
# Team number drives hostname (`acuity-NNNN.local`) and the
# FRC-standard static IP `10.TE.AM.11`.
TEAM=1279

# Robot WiFi SSID + password. Leave PSK blank for an open network.

# Country code for WiFi regulatory domain.

            setup wizard.

            skips the AP-mode wizard. You don't need
            acuity.conf at all in that case.
